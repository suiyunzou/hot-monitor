# 筛选器重构记录

更新时间：2026-06-08

## 背景

本次重构针对首页热点筛选器的两个核心问题：

- 页面默认展示条数过少，旧信息被隐藏，用户误以为历史信息不存在。
- 筛选条件过少且缺少执行反馈，无法判断当前筛选是否生效、失败原因和实际命中总数。

同时修复筛选器 UI 在展开后撑高布局、平铺按钮过多、默认值文案过长、日期按钮文字空白等体验问题。

## 当前实现

### 后端接口

`GET /api/analyze` 现在支持以下查询参数：

- `q`：标题、摘要、原因、来源、作者等文本搜索。
- `status`：`all | confirmed | multi-source | social | verify`。
- `source`：`all | official | search | social`。
- `sort`：`time | score | views | replies`。
- `minScore`：最低热度分，0-100。
- `minConfidence`：最低置信分，0-100。
- `limit`：返回条数，默认 30，最大 100。
- `startDate` / `endDate`：时间范围。

接口返回新增：

- `totalTopics`：当前筛选条件下的数据库命中总数。
- `filters.active`：当前是否有筛选条件生效。
- `filters.issues`：无效参数或回退原因。
- `filters.returnedTopics`：本次实际返回数量。
- `filters.resultLimit`：本次展示上限。
- `filters.optimized`：数据库 count、limit、sort 是否生效，以及指标排序候选集大小。

### 性能策略

- 时间、热度排序优先在数据库完成排序和分页。
- 观看量、评论数排序需要聚合来源互动指标，目前只取有限候选集后在 Node 内排序，避免随全量数据线性增长。
- 默认首页展示从 12 条提升到 30 条，用户可切换到 12 / 30 / 60 / 100 条。

### 前端筛选器

直接点击项：

- 状态：全部、已确认、多源线索、社交热议、待核验。
- 时间：1天、7天、30天、全部时间。

下拉项：

- 排序：新闻时间、热度、观看量、评论数。
- 筛选浮层：来源、热度阈值、置信阈值、展示条数。

筛选浮层采用紧凑图标按钮，不再直接显示“全部来源 / 全部热度 / 全部置信”等长默认文案。点击图标时打开原生下拉菜单，完整选项只在菜单中展示。筛选浮层使用绝对定位，不参与主筛选栏布局，因此展开后不会撑高筛选栏或导致左侧状态标签上方出现空白。

### 筛选信息

筛选器底部展示当前筛选摘要，例如：

```text
命中 103 条，展示 30 条 · 当前时间范围 · 全部来源 · 全部热度 · 全部置信 · 按新闻时间排序
```

当接口参数非法或请求失败时，会在同一区域展示失败原因。

## 已修复问题

- 日期按钮空白：原因是 `.date-filter button` 使用了 `font-size: 0`，已恢复为正常字号。
- 筛选展开后平铺过长：已改为紧凑图标式下拉触发器。
- 筛选面板撑高布局：已改为右侧浮层，不占据 `filter-dock` 的文档流高度。
- 默认展示太少：SSR 初始查询和接口默认返回均改为 30 条。
- 筛选是否生效不可见：新增命中总数、展示条数、筛选摘要、失败原因。

## 验证方式

已通过：

```powershell
npm run typecheck
npm run build
```

建议人工验收：

1. 打开首页，确认默认展示 30 条。
2. 切换 1天 / 7天 / 30天 / 全部时间，确认按钮文字可见且筛选摘要变化。
3. 点击“筛选”，确认浮层出现在按钮下方，且不撑高顶部筛选栏。
4. 点击来源、热度、置信、展示四个图标，确认下拉选项完整可见。
5. 切换筛选条件后，确认命中总数、展示数量和卡片列表同步变化。
6. 测试 `sort=views` / `sort=replies`，确认排序结果可用且不会拉取全量数据。

## 后续建议

- 指标排序当前仍需要 Node 内聚合有限候选集，数据继续增长后建议把来源互动指标汇总到 `HotTopic` 或独立统计表，支持数据库级排序。
- 搜索仍基于 `contains`，中文分词和语义相关性有限；后续可接 SQLite FTS5 或 embedding 检索。
- 筛选浮层目前使用原生 `select`，实现成本低且稳定；如果后续需要更强交互，可替换为无障碍菜单组件。
